# Beginner Coding Crash Course

Sessions are designed to run about one hour each. Study sessions are marked with **[ STUDY ]** — these are just as important as the content sessions. Don't skip them.

---

## Phase 0 — How Computers Think
*Goal: build a mental model before writing a single line of code.*

**Session 1 — KA: Computers and the Internet**
Go to **[Khan Academy: AP Computer Science Principles](https://www.khanacademy.org/computing/ap-computer-science-principles)** and open the "Computers and the Internet" unit. Watch all videos and complete the exercises. Pause and re-watch anything confusing — there's no timer.

> **Note on Khan Academy links:** All KA links in this plan point to the top of each course rather than individual units, because KA occasionally reorganizes their content and deep links can break. If a unit isn't immediately visible when you land on the page, scroll down to the course table of contents and find it by name.

**Session 2 — KA: Algorithms**
In the same AP CSP course, complete the "Algorithms" unit. These exercises have no coding — they're about how a computer breaks a problem into steps.

**Session 3 — [ STUDY ] Concepts check**
Close your notes. Write down in your own words: *What is a computer program? What is an algorithm?* Then open your notes and fill in what you missed. Writing it yourself is the test — if you can't explain it, you haven't learned it yet.

**Session 4 — CS50: Lecture 0**
Watch **[Lecture 0: Scratch](https://cs50.harvard.edu/x/weeks/0/)** (also on the **[CS50 YouTube channel](https://www.youtube.com/@cs50)**). Take notes by hand. Don't do any assignments — just watch.

**Session 5 — CS50: Lecture 1**
Watch **[Lecture 1: C](https://cs50.harvard.edu/x/weeks/1/)** (also on the **[CS50 YouTube channel](https://www.youtube.com/@cs50)**), pause when needed, take notes.

**Session 6 — [ STUDY ] Phase 0 wrap-up**
Without looking at anything, write a short paragraph about what you learned in Phase 0. What surprised you? What still feels fuzzy? Fuzziness is fine — write it down so you know where to pay attention in Phase 1.

*Goal: learn the core building blocks of programming in a forgiving environment.*

Go to **[Khan Academy: Intro to JS — Drawing and Animation](https://www.khanacademy.org/computing/computer-programming/programming)**. The canvas on the right updates live as you type, which makes it immediately obvious what your code is doing.

## Phase 2 — Learn Python
*Goal: learn a clean, professional language used in real jobs, data science, and research.*

Python reads almost like plain English, has minimal punctuation clutter, and is one of the most in-demand languages in the world. Download it from **[python.org](https://www.python.org)** — the installer includes **IDLE**, a simple built-in editor. Open IDLE from your Applications folder after installing.

The MIT course lives here: **[MIT OCW 6.0001 — Introduction to Computer Science and Programming in Python](https://ocw.mit.edu/courses/6-0001-introduction-to-computer-science-and-programming-in-python-fall-2016/)**

The pattern for most lecture sessions is the same: watch, take notes, then in the following practice session open IDLE and try to reproduce the examples from scratch without looking. That practice session is where the learning actually happens.

You'll also read chapters from **[Automate the Boring Stuff with Python](https://automatetheboringstuff.com)** (free online) alongside the lectures — it covers the same concepts but frames them around practical tasks, which helps when the MIT material feels abstract.

## Phase 4 — Build Something Real
*Goal: experience the full loop of having an idea, writing code to make it happen, and fixing it when it breaks.*

Pick **one** project and finish it completely:

- **Quiz game** — asks multiple-choice questions, tracks the score, prints a result at the end
- **Budget tracker** — user adds income and expenses, program shows a running total
- **Personal automation** — a script that does something useful in your own life (rename files, pull data from a spreadsheet, etc.)

**Session 56 — Planning**
Don't write any code yet. Write in plain English exactly what your program should do, step by step. Break it into the smallest pieces you can. This is the most skipped step and the one that saves the most time.

**Session 57 — Build: Simplest version**
Get the bare minimum working — one feature, no edge cases, no polish. It should actually run.

**Session 58 — Build: Second feature**
Add the next thing on your list from Session 56.

**Session 59 — Build: Third feature**
Keep going.

**Session 60 — Debugging session**
Fix what's broken. Read every error message fully before googling. The answer is usually in the message itself.

**Session 61 — Build: Final feature**
Add your last planned feature or refine what you have.

**Session 62 — Polish**
Try to break your own program. Enter weird inputs, skip expected steps, type letters where it expects numbers. Fix what you find.

**Session 63 — [ STUDY ] Reflection**
Write a paragraph: what did you build, how does it work, and what would you do differently if you started over? You don't need to share this with anyone — writing it forces you to think clearly about what you actually learned.

## What to Google When You're Stuck

This is a skill in itself. Knowing *how* to search is as important as knowing *what* to search.

**When you get an error message:**
Search the exact error text in quotes, plus the word `python`. Example:
> `"TypeError: can only concatenate str (not int) to str" python`

Don't paraphrase the error — copy it verbatim. Someone has almost certainly hit the exact same message.

**When you don't know how to do something:**
Use the pattern: `python how to [thing you want]`. Example:
> `python how to check if a number is even`
> `python how to loop through a list`

**When the docs feel too dense:**
Search the same topic with `"for beginners"` or `"simple example"` added. Example:
> `python dictionary simple example`

**When you want the authoritative answer:**
Go straight to **[docs.python.org/3](https://docs.python.org/3/)** — Python's official documentation. It's more readable than it looks. Use the search bar at the top.

**When a [Stack Overflow](https://stackoverflow.com) answer works but you don't understand why:**
Don't just copy it. Paste it into your file, then search for each piece of it you don't recognize. Understanding *why* it works is the rep.

**Terms worth knowing early (they'll come up constantly):**
- **[f-strings](https://docs.python.org/3/tutorial/inputoutput.html#formatted-string-literals)** — a cleaner way to put variables inside text
- **[list comprehensions](https://docs.python.org/3/tutorial/datastructures.html#list-comprehensions)** — a compact way to build lists
- **[try/except](https://docs.python.org/3/tutorial/errors.html)** — how to handle errors gracefully
- **[reading a stack trace](https://docs.python.org/3/library/traceback.html)** — how to interpret the error report Python gives you

**The golden rule:** if you've been stuck for more than 30 minutes and searches aren't helping, you're probably asking the wrong question. Step back and describe the problem out loud as if explaining it to someone else. Often that alone unsticks you — this is called [rubber duck debugging](https://en.wikipedia.org/wiki/Rubber_duck_debugging), and it genuinely works.

## Three habits that matter more than any course

1. **Write code every day, even for 20 minutes.** Consistency beats marathon sessions.
2. **Type everything out by hand.** Copy-pasting code you don't understand yet is skipping the rep.
3. **Get comfortable being stuck.** Every professional developer spends most of their day debugging. It's not a sign you're doing it wrong.
